# 审核手册

当你需要重建背景、判断设计、撰写问题，或把诊断继续推进成修正路径时，使用这份参考。

## 目标

先搞清楚这份设计到底想解决什么，再判断它做得好不好；并且把审核重心推向结构修正，而不是停在装饰性建议上。

## 背景重建

先推断，再确认这些信息：

- 页面或流程类型
- 产品类型
- 所处行业
- 目标用户
- 核心场景
- 业务目标
- 工作流或转化链路中的位置
- 当前假设
- 当前未知项

## 证据来源

优先用更强的证据：

1. 用户确认过的上下文
2. PRD、产品文档或研究资料
3. Figma 结构化数据
4. 截图或导出图
5. 模型推断

常见可用信号：

- 页面标题和可见文案
- CTA 文案
- 字段标签和表格列名
- 导航结构
- 信任或风险提示
- 状态标签
- 组件层级
- 原型链接
- 变量命名和 token 使用

当证据只有截图，或材料明显不完整时：

- 下调置信度，不要假装精确
- 明确写出证据边界
- 审核只针对真正看得到的内容展开

## 确认回合

第一次背景重建后，再确认或修正这些判断：

- 这份设计服务谁
- 用户此刻在尝试完成什么任务
- 这条设计想推动什么业务结果
- 草稿里看不到、但真实存在的约束或行业规则

---

## 审核框架

设计审核分为两个互补的层次：

| 层次 | 定位 | 解决的问题 |
|------|------|-----------|
| **三流一致性检查** | 跨页面/跨步骤的连贯性检查 | "单页都对，但整条链断了" |
| **五维度质量检查** | 单页/单模块的质量检查 | "这一页本身做得怎么样" |

**职责分工规则**：同一个问题只在一个体系中出现。如果问题发生在页面与页面的衔接处，归三流；如果问题发生在单页内部，归五维度。

---

## 三流一致性检查

三流检查的核心任务是：抓住那种"单页看都没错，但整条链就是断的"问题。

至少选一条关键任务链，显式检查这三条流是否保持一致。复杂流程可以看 2 到 3 条关键路径。除非用户明确要求，不要把每一页都做成穷举式审计。

### 旅程流 — 服务设计方法论

旅程流关注的是：从用户进来到任务完成，整条大链路是否成立。

| 检查项 | 方法论来源 | 判断标准 |
|--------|-----------|---------|
| 任务路径完整性 | 用户旅程地图 | 从入口到完成，是否每一步都有明确的下一步可走 |
| 触点一致性 | 服务蓝图 | 跨页面的状态、信息、语调是否连贯无矛盾 |
| 关键时刻保障 | Moment of Truth | 高风险节点（付费/授权/不可逆操作）是否有充分信息支撑决策 |
| 异常路径兜底 | 服务蓝图后台线 | 出错、中断、回退时是否有恢复路径 |
| 端到端效率 | JTBD | 完成核心任务是否存在多余步骤或重复操作 |

**重点观察**：
- 用户能不能一直往前走，还是会在某一步卡住不知道下一步是什么
- 前后页面说的是不是同一个故事，有没有信息打架
- 关键决策点（比如付款、授权）信息够不够做判断
- 出了问题有没有路回来

### 操作流 — HCI/交互设计方法论

操作流关注的是：当前每一步动作和系统反馈是否清楚、连续。

| 检查项 | 方法论来源 | 判断标准 |
|--------|-----------|---------|
| 系统状态可见性 | Nielsen 启发式 #1 | 用户能否随时知道"我在哪、系统在做什么" |
| 操作-反馈对应 | Nielsen 启发式 #1 | 每个操作后是否有即时、清晰、对应的反馈 |
| 一致性与标准 | Nielsen 启发式 #4 | 同类交互模式是否前后统一、符合平台惯例 |
| 容错与可逆 | Nielsen 启发式 #5, #9 | 操作失误后能否撤销，错误恢复路径是否清晰 |
| 选择复杂度 | Hick 定律 | 单步决策选项数量和认知负担是否适度 |
| 操作目标可达 | Fitts 定律 | 关键操作的位置、大小、视觉突出度是否合理 |

**重点观察**：
- 用户是不是随时知道系统在干嘛、自己在哪一步
- 点了一个按钮，有没有马上告诉他发生了什么
- 同样的操作在不同页面表现是不是一样
- 做错了能不能挽救

### 心智流 — 认知心理学方法论

心智流关注的是：用户理解、确定感和信任感是否在往前走，而不是中途崩掉。

| 检查项 | 方法论来源 | 判断标准 |
|--------|-----------|---------|
| 心智模型对齐 | Norman 心智模型理论 | 系统概念是否匹配用户已有的理解方式 |
| 认知负荷控制 | Sweller 认知负荷理论 | 信息密度、术语复杂度是否在可接受范围 |
| 预期管理 | 可供性理论 (Affordance) | 用户对"点了会怎样"的预期是否被正确设定 |
| 信任递进 | 信任信号理论 | 从犹豫到确信的信任感是否随操作逐步建立 |
| 可发现性 | Norman 设计原则 | 功能是否无需说明就能被发现和理解 |
| 情绪曲线 | 峰终定律 (Peak-End Rule) | 关键节点的用户情绪是否被合理照顾 |

**重点观察**：
- 用户是不是越走越明白，还是越走越糊涂
- 用户对"接下来会发生什么"有没有准确预期
- 用户对这个产品的信任感是在建立还是在崩塌
- 关键节点有没有让用户觉得"被照顾到了"

### 三流检查的输出要求

在人类 HTML 报告里，优先用"对齐真实阶段的纵向时间轴"，而不是三张抽象长卡。每个节点只写真正存在的问题块，每个问题块都用"问题 + 解法"的人话表达。

三条流要观察它们是在互相支撑，还是互相打架：
- 断裂发生在哪一步
- 断的是哪一层（旅程/操作/心智）
- 设计是不是每一页都局部正确，但拼起来却不是一条能走通的任务链

## 五维度质量检查

五维度检查的定位是单页/单模块质量检查，与三流互补。每个维度都有具体的检查标准和方法论支撑。

### 1. 场景匹配

检查这一页是否真正理解并服务了它声称要服务的场景。

| 检查项 | 方法论来源 | 判断标准 |
|--------|-----------|---------|
| 用户画像验证 | 用户画像 (Persona) | 页面内容是否真正针对目标用户群体，而不是泛化的"所有人" |
| 场景覆盖度 | 场景设计 (Scenario Design) | 主场景是否清晰覆盖，重要的变体场景有没有被忽略 |
| 边缘用户考量 | 包容性设计 | 非典型用户（新手、高龄、非核心用户）能否顺利使用 |
| 场景-功能对应 | 任务分析 | 页面功能是否直接回应场景中用户想做的事 |

### 2. 认知与操作成本

检查用户完成任务需要付出的脑力和操作代价是否合理。

| 检查项 | 方法论来源 | 判断标准 |
|--------|-----------|---------|
| 内在认知负荷 | Sweller 认知负荷理论 | 任务本身的复杂度有没有被合理拆解 |
| 外在认知负荷 | Sweller 认知负荷理论 | 界面呈现有没有增加不必要的理解难度 |
| 关联认知负荷 | Sweller 认知负荷理论 | 信息组织是否帮助用户建立有效的心智模型 |
| 工作记忆限制 | Miller 7±2 法则 | 单屏需要用户同时记住的信息是否超过 5-9 项 |

### 3. 用户心智与习惯

检查设计是否尊重用户已经形成的认知习惯。

| 检查项 | 方法论来源 | 判断标准 |
|--------|-----------|---------|
| 跨产品习惯 | Jakob 定律 | 用户在其他同类产品上养成的习惯是否被尊重 |
| 概念映射 | Norman 心智模型理论 | 产品概念是否匹配用户已有的理解方式 |
| 术语熟悉度 | 认知流畅性 | 使用的词汇是用户自然会用的，还是需要学习的专业术语 |
| 交互惯例 | 平台设计规范 | 常见交互模式是否符合平台惯例（iOS/Android/Web） |

### 4. 价值承接

检查用户在这一页能否清晰感知到价值，并被顺畅地引导到下一步。

| 检查项 | 方法论来源 | 判断标准 |
|--------|-----------|---------|
| 价值主张清晰度 | 价值主张画布 | 用户能否在 5 秒内理解"这对我有什么用" |
| 功能可见性 | Affordance 理论 | 核心功能是否显而易见，不需要用户去找 |
| 功能可理解性 | Affordance 理论 | 看到功能后，用户是否知道它能做什么、怎么用 |
| 价值连续性 | 用户旅程设计 | 上一步承诺的价值在这一步是否被兑现 |

### 5. 基础体验

检查基础的交互质量和异常处理。

| 检查项 | 方法论来源 | 判断标准 |
|--------|-----------|---------|
| 错误预防 | Nielsen 启发式 #5 | 设计是否从一开始就降低了出错可能性 |
| 错误处理 | Nielsen 启发式 #9 | 出错后提示是否清楚，是否告诉用户怎么修复 |
| 帮助与文档 | Nielsen 启发式 #10 | 复杂任务是否有适当的帮助入口 |
| 使用灵活性 | Nielsen 启发式 #7 | 高级用户是否有更高效的完成方式 |

## 检查执行规则

### 逐项检查

LLM 评审时必须逐项过一遍检查清单。每项可标记为：

- ✓ 适用并检查 — 该检查项适用于当前设计，已完成检查
- — 不适用（附理由）— 该检查项不适用于当前场景，说明为什么
- ? 信息不足无法判断 — 缺少必要信息，标注缺什么

### 问题归因

发现的问题必须标注对应的方法论来源，便于设计师理解问题本质。例如：
- "违反 Nielsen #4 一致性与标准：前一页用『提交』，这一页用『确认』，同样的动作用了不同的词"
- "违反 Hick 定律：首页同时展示 12 个入口，用户很难快速决定点哪个"

### 方法论是内在支撑，输出是人话

**关键**：方法论是你判断问题的内在逻辑支撑，但输出报告仍然要用业务语言"说人话"。不能让设计师觉得在考试。

**不好的写法**（像考试）：
> 此页面违反了 Sweller 认知负荷理论中的外在负荷控制原则，信息密度过高导致用户工作记忆超载。

**好的写法**（说人话）：
> 这一页把 15 项权限说明堆在一起，用户根本看不过来，更别说做出判断了。建议分组呈现，先让用户看到最关键的 2-3 项。
> （方法论支撑：外在认知负荷过高）

方法论标注可以用括号附在段落末尾，或者在问题卡片的元信息区域显示，但不要喧宾夺主。

### 三流时间轴要求

三流时间轴报告中每个节点必须关联具体检查项。不是写"旅程流有问题"，而是直接写方法论名称，如"任务路径完整性：第三步到第四步之间没有明确引导"。不要在方法论名称前加"旅程流 >"、"操作流 >"、"心智流 >"这样的前缀。

### 方法论标注的使用范围

方法论来源标注（methodology_source）应在以下位置使用：

1. **问题卡片中**：每个 issue 必须标注关联的方法论原则
2. **三流时间轴中**：每个有问题的阶段标注关联的检查项和方法论
3. **修改建议中**：建议的理论依据

标注格式保持简洁，如 "Nielsen #4"、"Sweller 认知负荷"、"峰终定律"。不要在方法论名称前加"旅程流 >"、"操作流 >"、"心智流 >"等三流层级前缀。直接写方法论名称即可。

标注是为了让诊断有据可循，不是为了显得专业。如果某个问题的发现不依赖特定方法论，可以不标注。但对于大多数 UX 问题，可关联到一条具体方法论原则会让诊断更有说服力。

## 必跑检查

至少在最重要的任务路径上跑这些检查：

- `认知与文案负担检查`
- `three-flow continuity review`，也就是旅程流、操作流、心智流的一致性检查
- `场景压力测试`，针对最关键的失败或压力场景
- 当文案、图标语义、状态覆盖重要时，再补 `微观细节检查`

## 人类问题结构

每个问题既要能支撑设计师阅读，也要能支撑机器追踪。

必须同时保留：

- 用于追踪和 diff 的 `stable_id`
- 方便人看的显示编号，比如 `ISSUE-001`

每个问题至少要覆盖这些信息：

- `title`
- `category`
- `severity`
- `confidence`
- `evidence_ids` 或等价证据依据
- `methodology_source` — 该问题违反了哪条方法论原则
- `what_i_see`
- `why_it_is_a_problem`
- `what_misunderstanding_it_reveals`
- `who_it_hurts`
- `likely_consequence`
- `recommended_direction`
- `discussion_prompts`

## 人类写作规则

机器对象可以保持结构化，但 HTML 报告绝不能写成 schema 原样外露的样子。

### 标题规则

问题标题先服务于人读懂，再服务于追踪。

- 标题要让设计师第一眼就能明白
- 优先写"具体对象 + 具体问题 + 可见后果"
- 中文里优先自然的主谓宾，而不是压缩的抽象名词堆叠
- 如果一个标题必须先解释，才变得能读懂，那它就该重写
- `stable_id` 负责机器精确，不要逼标题同时承担机器工作

避免这类标题：

- `首步路径选择依赖用户自诊断账号状态`
- `分流策略与用户心智存在错位`
- `认知负担过重`

优先写成：

- `用户可能不知道自己是否注册过万豪，却必须先选开通还是绑定`
- `页面把业务分流直接丢给用户判断，选错后要到失败页才发现`
- `协议页把必选授权和营销偏好混在一起，用户很难一眼分清`

### 段落规则

对人类报告来说：

- 不要默认把 `what_i_see`、`who_it_hurts`、`likely_consequence`、`discussion_prompts` 这些字段名直接露出来
- 把它们合并成一个有因果关系的诊断段落，讲清楚发生了什么、为什么重要、影响谁
- 再补一个修正段落，说明应该怎么改、为什么这么改
- 方法论来源可以用括号附在诊断段落末尾，帮助设计师理解问题本质
- 当讨论问题会实质影响建议时，把它压成一个短的 `需要确认` 段落
- 证据要可见，但不要让 node ID 和工具步骤统治阅读节奏

### 中文写作规则

当报告语言是 `zh-CN`：

- 写得像一个中文语境下的强设计负责人，而不是英文分析文的翻译稿
- 优先短而完整的句子，不要堆太多修饰语
- 优先用户能理解的事实，不要沉迷内部术语
- 抽象 UX 词要落到可见行为上去解释
- 术语只在真有帮助时用，不要拿术语顶替解释
- 人类报告里不要出现 `结构包`、`整改包` 这类 PM 腔

例如：

- 不写 `用户自诊断账号状态`，改写成 `用户得先自己判断自己到底算未注册还是已注册`
- 不写 `路径分流`，改写成 `先选开通还是绑定`
- 不写 `认知负担`，改写成 `用户得自己多想一步，而且很容易想错`

## 审核维度（兼容旧引用）

从这些维度看设计：

1. `问题与价值`
2. `用户与洞察`
3. `场景匹配`
4. `用户心智与习惯`
5. `认知与操作成本`
6. `行业基线与替代路径`
7. `基础体验检查`

## 严重度参考

- `critical`：直接挡住任务完成、产生严重信任风险，或暴露战略级判断错误
- `high`：大概率造成失败、困惑、放弃，或带来高成本返工
- `medium`：削弱理解、效率或信心，但不至于完全卡死任务
- `low`：能感觉到问题，但不是结果的核心决定因素

## 置信度参考

- `high`：有精确文案或节点证据，并且截图也确认了
- `medium`：可见、结构上也说得通，但没有精确到节点级的证据支撑
- `low`：只能从局部视觉、稀疏读取或工具边界里推断

## 稳定问题标识

- 内部逻辑必须依赖 `stable_id`，不要依赖显示编号
- `stable_id` 应该是混合格式：可读语义片段 + 稳定后缀
- 显示编号可以在复跑后重排
- 仅仅因为显示顺序变化，`stable_id` 不应变化
- 持续存在的问题、已解决的问题、发生实质变化的问题，都应通过 `stable_id` 追踪

## 修正顺序

针对用户选中的问题，按这个顺序推进：

1. 重述真正的问题
2. 判断最可能的根因
3. 给出 2 到 3 条修正路径
4. 解释取舍
5. 推荐一个方向
6. 定义下一轮要验证什么

## 根因追问

问清楚到底是哪一层断了：

- 问题定义错了
- 用户假设错了
- 场景假设错了
- 信息层级错了
- 先后顺序错了
- 交互契约错了
- 缺少状态或恢复路径
- 文案或术语漂移了

## 审核姿态

- 默认从产品和体验合理性出发，不从风格好不好看出发
