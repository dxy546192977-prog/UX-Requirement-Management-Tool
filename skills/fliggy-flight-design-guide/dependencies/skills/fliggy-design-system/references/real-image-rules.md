# 真实图片使用规则

## 目标

- 页面在需要图片时优先展示真实照片，不用空白占位、灰块占位或无关素材
- 图片要服务业务表达，而不是只补“设计感”
- 选图要和页面类型、模块语义、目的地和商品类型一致

## 什么时候应该有图

- 详情页：通常需要头图、相册、商品卡图片
- 列表页：通常需要列表缩略图、卡片图
- 频道页：通常需要主题图、运营卡图、目的地图
- AI 推荐页：在商品推荐卡、目的地卡、路线卡里可以带图
- 下单页：默认不需要大图 hero，只有在摘要卡本身明确需要缩略图时才补一张轻量图片

## 什么时候不应该强行补图

- 填写订单页顶部
- 支付确认页
- 纯表单页
- 纯结果说明页

这些页面的重点是确认、填写、支付、决策，不是内容浏览。

## 业务场景与图片类型映射

- 酒店：酒店外观、大堂、客房、泳池、餐厅、窗景
- 民宿：院子、客厅、床铺、露台、山景或海景
- 门票：景点地标、乐园城堡、雪场、演出现场、博物馆展陈
- 机票：机场航站楼、机舱窗景、值机柜台、出发大厅、目的地城市
- 度假线路：海岛、沙滩、泳池、山景、公路、露营地
- 美食：菜品、餐厅环境、吧台、露台座位

## 搜图方法

### 搜图公式

优先把检索词组织成：

`品类 + 细分场景 + 城市/目的地 + 镜头类型 + 氛围词`

例如：

- `luxury hotel room shanghai night view`
- `family resort hotel sanya pool`
- `theme park castle fireworks night`
- `airport terminal business travel`
- `ski resort cable car snow mountain`

### 多平台搜图策略

- 有联网能力时，优先从 Pexels、Unsplash 等真实图库取图
- 如果用户提供了 Pexels API key，请求时把 key 直接放在 `Authorization` 请求头，不要加 `Bearer`
- 如果有多个图库可选，优先选返回结果更贴近当前业务实体的那一个，而不是只看图片清晰度
- 结果优先选择横向大图做头图，选择中近景做列表卡和商品卡
- 如果同一页有多张图，尽量保证摄影风格一致，不要一张偏纪实、一张偏极简棚拍、一张偏插画风

### 搜索词优化原则

- 强实体词优先：品牌、景点名、演出名、城市名优先放前面
- 再补业务词：hotel、ticket、airport、resort、museum、show
- 再补模块词：hero、room、lobby、night、exterior、interior、fireworks、parade
- 迪士尼门票页这类强场景页面，优先搜索 `Disneyland`、`Disney castle`、`Disney parade`、`Disney fireworks` 等实体词组合
- 不要只搜大类词，例如门票页不要只搜 `travel` 或 `theme park`

### 相关性自检

选图后至少检查下面几项：

- 这张图是否真的是当前业务品类
- 这张图是否与当前模块位置匹配
- 这张图是否和标题、价格、CTA 语义一致
- 这张图是否避免误导用户

## 前端原型里的推荐实现

- 优先封装一个 `getSceneImage` 或 `resolveSceneImage` 层，集中处理场景到图片的映射
- 如果运行时可请求 Pexels，可在这个层里调用图片接口
- 如果当前环境没有联网条件，可退化到一组预选真实图片 URL
- 无论哪种方式，都不要把用户的 API key 硬编码进源文件

## 最低交付标准

- 需要图的页面必须有真实图片
- 图片必须与业务场景有关
- 头图、缩略图、商品图的角色要清楚
- 不能用无意义占位图替代最终效果
